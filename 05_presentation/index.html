Requêtes SQL 5 requêtes analysées

1
Joueurs avec un triple double — saison 2024/25
✓ Optimisée ▶

Sélectionne les joueurs ayant au moins 3 catégories statistiques ≥ 10 en moyenne sur la saison.

SELECT p.player_name, ps.season,
       ROUND(AVG(ps.pts), 1) AS moy_points,
       ROUND(AVG(ps.reb), 1) AS moy_rebonds,
       ROUND(AVG(ps.ast), 1) AS moy_passes,
       ROUND(AVG(ps.blk), 1) AS moy_contres,
       ROUND(AVG(ps.stl), 1) AS moy_interceptions
FROM player_stats ps
JOIN players p ON ps.player_id = p.player_id
WHERE ps.season = '2024-25'
GROUP BY p.player_name, ps.season
HAVING
  (CASE WHEN AVG(ps.pts) >= 10 THEN 1 ELSE 0 END
  +CASE WHEN AVG(ps.reb) >= 10 THEN 1 ELSE 0 END
  +CASE WHEN AVG(ps.ast) >= 10 THEN 1 ELSE 0 END
  +CASE WHEN AVG(ps.blk) >= 10 THEN 1 ELSE 0 END
  +CASE WHEN AVG(ps.stl) >= 10 THEN 1 ELSE 0 END) >= 3
ORDER BY moy_points DESC;

2
Joueurs > 2m ayant fait des interceptions
✓ Optimisée ▶

Utilise une sous-requête EXISTS pour vérifier l'existence de matchs avec interceptions.

SELECT DISTINCT
    p.player_name, p.height_cm,
    COUNT(DISTINCT ps.game_id) AS nb_matchs
FROM players p
JOIN player_stats ps ON p.player_id = ps.player_id
WHERE p.height_cm > 200
  AND EXISTS (
      SELECT 1 FROM player_stats ps2
      WHERE ps2.player_id = p.player_id
        AND ps2.stl > 0
  )
GROUP BY p.player_name, p.height_cm
ORDER BY p.height_cm DESC;
3
Total des rebonds — Golden State Warriors
✓ Optimisée ▶

Jointure triple : équipes → joueurs → statistiques. Filtre sur le nom d'équipe et la saison.

SELECT t.team_name, SUM(ps.reb) AS total_rebonds
FROM teams t
JOIN players p    ON p.team_id   = t.team_id
JOIN player_stats ps ON ps.player_id = p.player_id
WHERE t.team_name LIKE '%Golden State%'
  AND ps.season = '2024-25'
GROUP BY t.team_name;
4
Maximum de points dans un seul match
✓ Optimisée ▶

Sous-requête scalaire pour MAX — l'index B-tree sur pts permet de lire directement la feuille la plus à droite de l'arbre.

SELECT p.player_name, g.game_date, ps.pts
FROM player_stats ps
JOIN players p ON ps.player_id = p.player_id
JOIN games g   ON ps.game_id   = g.game_id
WHERE ps.pts = (
    SELECT MAX(pts) FROM player_stats
)
ORDER BY g.game_date;
5
Joueur avec le plus de points — saison 2024/25
✓ Optimisée ▶

HAVING avec MAX imbriqué sur un GROUP BY — nécessite les index créés aux étapes précédentes.

SELECT p.player_name, ps.season,
       SUM(ps.pts) AS total_points
FROM player_stats ps
JOIN players p ON ps.player_id = p.player_id
WHERE ps.season = '2024-25'
GROUP BY p.player_name, ps.season
HAVING SUM(ps.pts) = (
    SELECT MAX(SUM(ps2.pts))
    FROM player_stats ps2
    WHERE ps2.season = '2024-25'
    GROUP BY ps2.player_id
);

Index créés Justification et types

Nom de l'index Table Colonnes Type Justification
idx_ps_season_player player_stats season, player_id Composé B-tree Filtre saison + jointure joueur en une seule passe (covering index)
idx_ps_pts player_stats pts B-tree MAX(pts) lit la feuille la plus à droite sans scanner la table
idx_players_height players height_cm B-tree INDEX RANGE SCAN pour la condition height_cm > 200
idx_ps_player_stl player_stats player_id, stl Composé B-tree Covering index pour la sous-requête EXISTS (pas d'accès table)
idx_teams_name teams team_name B-tree Accès rapide pour la condition LIKE '%Golden State%'
idx_players_team players team_id B-tree Optimise la jointure players → teams
idx_ps_game player_stats game_id B-tree Jointure player_stats → games (R4)
idx_bitmap_season player_stats season Bitmap Peu de valeurs distinctes (~5 saisons) → opérations bit à bit très rapides

Bilan d'optimisation Checklist

Aucun TABLE ACCESS FULL après optimisation

Toutes les requêtes utilisent INDEX RANGE SCAN ou INDEX UNIQUE SCAN après création des index.

Index composés = covering indexes

Pour R1 et R2, Oracle accède uniquement aux blocs d'index, sans lire la table physique.

Index Bitmap sur la colonne season

La saison a peu de valeurs distinctes (~5) : le Bitmap index est idéal pour les filtres sur cette colonne.

MAX optimisé via B-tree (R4)

Un index B-tree sur pts permet à Oracle de lire directement la feuille la plus à droite de l'arbre sans tri.

Inconvénient : maintenance des index

Chaque INSERT/UPDATE/DELETE sur player_stats doit aussi mettre à jour les 8 index créés. Pour des chargements massifs, désactiver les index temporairement est recommandé.
